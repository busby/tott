TDD

Test-driven development is a short, repetitive process.

  1. Write a failing test case for an enhancement (red)
  2. Implement the minimum code to pass the test (green)
  3. Refactor the code to meet project standards (refactor)

Use Cases

  • Focus on requirements
  • Continuous integration
  • Development consistency

But what do you test?

BDD

Behavior-driven development builds on TDD.

  1. Specify desired behavior in a consistent format
  2. Write a failing test case for a scenario
  3. Implement the minimum code
  4. Refactor the code to meet standards

But what is a consistent format?

One Format: User Stories

Story: App store purchases require a credit card

In order to charge customers
As an app store owner
I want to ensure the customer has a valid cc on file

Scenario 1: App store should prompt for missing cc info
Given a customer attempts a purchase
And the customer has no cc on file
When I attempt to charge her cc
Then I should prompt for her cc info
And I should validate the cc info

Secnario 2: Expired cc info should be purged
...
